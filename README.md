# triton 3.0 — re-identification тритонов по узору брюшка

Open-set fine-grained re-ID: фото брюшка тритона → какая это особь из базы (с калиброванным
score и наглядным обоснованием) или новая особь. ВКР магистратуры «Прикладной ИИ» (ТГУ);
данные предоставлены ИПЭЭ РАН им. Северцова.

Проект завершён: реализованы все 6 блоков (данные → сегментация и кроп → распрямление →
эмбеддер → матчинг пятен → гибрид и демо); финальная оценка — на однократно вскрытом sealed-test.

## Архитектура

Кроп **только брюшка** (YOLO seg + pose) → ориентация (голова/хвост) → **распрямление ленты**
(`unroll_ribbon`) → zero-shot **MegaDescriptor-L-384**, косинусный поиск top-K (numpy; FAISS — задел для масштабирования) →
матчинг созвездия центроидов пятен — даёт **оверлей совпавших пятен** (интерпретируемое
обоснование решения) → калиброванный %-score + решение known/new.

Диаграммы (use case, component, deployment, sequence и др.) — в `docs_public/uml/`.

## Установка

```bash
pip install -r requirements.txt   # зависимости (версии зафиксированы под проверенное окружение)
pip install -e .                  # сам пакет (editable); зависимости НЕ ставит — намеренно, см. комментарий в pyproject.toml
pip install -e ".[crop]"          # опционально (Блок 2): ultralytics / scikit-image — сегментация и кроп
```

Веса в репозиторий не входят: MegaDescriptor-L-384 загружается с HF Hub при первом запуске;
обученные веса YOLO seg/pose (~6.5/6.1 МБ) будут выложены отдельно (GitHub Releases / HF Hub)
после публикации.

## Демонстрация

Демо (`streamlit run app/demo.py`, вкладка «Опознание»): загрузка фотографии брюшка — система
выделяет и распрямляет узор, ищет ближайших особей и показывает результат.

- **Вход:** фотография брюшка (JPG/PNG).
- **Выход:** ранжированный список top-K кандидатов с оценкой сходства (0–100 %), решение
  «известная / новая особь» и **оверлей совпавших пятен** как наглядное обоснование.
- **Пример:** запрос → особь `PW-005`, уверенность 99 %; оверлей — 15 совпавших пятен (доля 75 %).

Скриншоты демонстрации содержат реальные кропы брюшек особей (данные заказчика) и в публичный
репозиторий не включаются. Схема взаимодействия и потока данных — в `docs_public/uml/`
(use case, sequence, component); демонстрация воспроизводится локально командой выше.

## Результаты (sealed-test, kpi_core)

Финальная система — zero-shot MegaDescriptor-L-384 + `unroll_ribbon`, зафиксирована на dev
**до** вскрытия test; sealed-test вскрыт однократно, порог open-set калиброван на dev, не на test.
Источник чисел — `artifacts/ab_test_headline.json`.

| Срез | n | recall@1 | recall@5 |
|---|---|---|---|
| overall | 112 | 0.250 | 0.446 |
| PW (ребристый тритон) | 11 | 0.818 | 1.000 |
| TK (тритон Карелина, temporal) | 101 | 0.188 | 0.386 |

- Галерея — 238 изображений; open-set AUROC 0.446 (known 112 / new 21).
- Срезы PW и TK несопоставимы между собой: PW — random/same-session enrollment,
  TK — строгий temporal-протокол (перепоимки между сессиями).
- KPI проекта (top-1 ≥ 75 % И top-5 ≥ 95 %) на overall не достигнут; узкое место —
  temporal-срез TK: тот же конвейер на PW планку проходит, т.е. ограничение в данных
  (воспроизводимость узора между сессиями), а не в методе.
- Свод чисел — `docs_public/ИТОГОВЫЕ_ЧИСЛА.md`, методологические оговорки —
  `docs_public/МЕТОДОЛОГИЯ_оговорки.md`.
- Время отклика — около 0,6 с на фотографию (CPU/MPS, офлайн). Профильная метрика качества
  для re-ID — recall@k (CMC) по особям, а не precision/FPS детекции.

## Принципы

- Единый канонический ридер изображений (`imageio.py`) — один препроцессинг везде.
- md5-дедуп **до** сплита (защита от утечки gallery↔probe).
- Детерминированные сплиты (seed=42); метрики temporal и random — раздельно.
- Sealed-test вскрывается один раз; повторное вскрытие и тюнинг на test запрещены гейтами CLI.
